# 项目长期规则

- 每次开始工作前先阅读本目录下的 `project_rules.md`、`project_framework.md`、`todolist.md`、`project_iteration_log.md`。
-始终保持大局观，优先保证整条业务链路稳定，不只修单点问题。
- 任何修改都必须以“不破坏现有全部功能”为前提，先兼容、后优化。
- 每次完成改动后，必须先做真人模拟测试，再结束本轮工作。
- 真人模拟测试应尽量贴近真实使用路径，重点覆盖网页中的实际操作流程。
- 完成修改并确认功能无误后，默认执行推送部署，不额外等待提醒。
- 目标是 1:1 复刻压缩包中的狼人杀发牌助手，所有现有功能必须保持可用，界面风格与核心交互保持一致。
- 修改时优先保证跨设备同步、发牌、房间加入、法官面板、电子法官、角色介绍、历史记录等完整链路不被破坏。
- 项目主要运行场景是手机端微信内打开，任何新增调整都要优先兼容窄屏、触控操作、底部安全区和固定面板遮挡问题。
- 发布目标是 GitHub Pages，所有静态资源路径、构建配置和部署流程都要以仓库公开访问为前提验证。
- 涉及联网能力时优先保持现有 Supabase 行为一致，不随意替换服务端数据结构、订阅方式或密钥配置。
- 每次动手前先更新 `todolist.md` 与 `project_iteration_log.md` 的本次目标和当前状态；完成后再补齐结果、遗留项和下一步。
- 如发现可以提升稳定性、移动端体验或部署可靠性的优化项，可以执行，但前提是不改变用户可见的原有产品定位和主要视觉风格。
- 每次动手前，先更新 todolist 和项目迭代记录中的“本次目标/当前状态/待办事项”，必要时同步更新项目框架。
- 每次改动完成后，必须再次更新项目框架、todolist、项目迭代记录，写清本次改动、当前结构变化、遗留事项和下一步建议。
- 这些文件的目标是减少重复通读代码、避免遗忘已完成事项和用户已交代的长期要求。
- 一般使用 git 维护迭代记录，代码改完并验证无误后及时提交，提交前提仍是保证现有功能完整可用。
